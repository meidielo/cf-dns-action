# Spike: ESM migration for v1.1.0

**Status:** Not started
**Owner:** @meidielo
**Target:** Answer the feasibility question before committing to v1.1.0 scope.

## Why this spike exists

Three deferred dep bumps are all gated on the same underlying change:
- `@actions/core@3` — ESM-only
- `zod@4` — breaking API changes, ESM-first packaging
- `eslint@9` family — already migrated (v1.0.x cleanup)

Before planning v1.1.0 / v1.2.0 / v1.3.0 as staged releases, prove the ESM path works end-to-end on a real GitHub Actions runner. If it doesn't, scope changes materially.

## The unknowns

1. **Does `@vercel/ncc --esm` produce a bundle that GitHub Actions can execute?**
   GitHub's action metadata (`action.yml`) declares `runs.using: 'node20'` and `runs.main: 'dist/index.js'`. Node 20 supports ESM, but the Actions runner's entrypoint loader has its own behavior — unconfirmed whether it imports the main file as ESM when the bundle is ESM.

2. **Does `package.json` need `"type": "module"`?**
   If yes, it ripples through the whole project — `require()` calls in tests/configs break. If the ncc ESM bundle is self-contained and the source tree stays CJS-interop via `.mjs` and `.cjs` suffixes, the ripple is smaller.

3. **Does `require.main === module` (if present anywhere) still work?**
   Legacy CJS idiom. ESM uses `import.meta.url` instead. Grep the source before migration.

4. **Do vitest and ncc still cooperate in ESM mode?**
   Vitest is ESM-native, no risk there. ncc's ESM output has historically had rough edges — need to verify.

## Acceptance criteria (spike output)

The spike is complete when this issue contains:
- [ ] A yes/no on whether `ncc --esm` output loads correctly as an action on `ubuntu-latest` runner.
- [ ] A confirmed list of source changes required (e.g., `require.main` replacements, `__dirname`/`__filename` replacements).
- [ ] A decision on `"type": "module"` in `package.json` — and if yes, the blast radius (which files break, which configs need renaming).
- [ ] A recommendation: proceed with staged releases (v1.1.0 ESM → v1.2.0 @actions/core → v1.3.0 zod) OR bundle into a single v1.1.0 OR defer indefinitely.

## Proposed method (15-min spike)

1. Create a throwaway branch `spike/esm`.
2. Add `"type": "module"` to `package.json` and update ncc command to `ncc build src/main.ts -o dist --source-map --target es2022 --esm`.
3. Run `npm run build`. Observe whether ncc emits a working ESM bundle.
4. Push the branch and create a workflow-dispatch job that invokes the action with dry-run inputs against a throwaway Cloudflare zone + Netlify site (or use existing test fixtures).
5. Capture:
   - Does the runner load the bundle?
   - Any deprecation warnings?
   - Any path-resolution errors (common with `__dirname` in ESM)?
6. Revert the branch without merging. Spike output lives in this issue.

## Non-goals (explicitly out of scope for the spike)

- Actually shipping v1.1.0.
- Migrating zod v3 → v4 — that's its own scope item.
- Bumping @actions/core — only relevant after ESM path is proven.
- Changing the action runtime from `node20` to `node24`.

## Risks if we skip this spike

- Committing to "v1.1.0 = ESM migration" without knowing if it works means the v1.x line stalls if the runner doesn't cooperate.
- Bundling all three breaking changes into a single release (which is what my original plan was) multiplies the debugging surface if anything regresses post-release.
